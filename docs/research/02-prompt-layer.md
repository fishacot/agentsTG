# От SOUL.md до Runtime Approval: Эталонная архитектура системных промптов для мультиагентных систем

## Архитектура системных запросов: Слоистый подход в OpenClaw и Manus

Анализ современных архитектур мультиагентных систем, таких как OpenClaw и Manus, показывает четкий тренд на декомпозицию сложных системных инструкций на управляемые, модульные слои. Этот подход позволяет гибко управлять поведением агентов, разделяя общую идентичность, правила взаимодействия, политики использования инструментов и контекст задачи. Такой слоистый подход является фундаментальной основой для построения надежных и масштабируемых агентных систем, позволяя одновременно сохранять общую цель и адаптироваться к специфическим условиям выполнения задачи.

OpenClaw представляет наиболее формализованный и документированный пример такого подхода. Его система промптов строится на основе нескольких ключевых файлов, которые вместе составляют полный контекст для LLM на каждом шаге диалога [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Эти файлы включают `SOUL.md`, `IDENTITY.md`, `AGENTS.md`, `TOOLS.md`, `USER.md` и `MEMORY.md`. Каждый из этих компонентов выполняет свою уникальную функцию, создавая целостную картину для агента. `SOUL.md` служит ядром агента, задавая его базовые принципы и философию, что делает его аналогом «конституции» или «души» [[11](https://arxiv.org/html/2603.10060v1), [12](https://arxiv.org/pdf/2603.10060)]. В нем могут быть зафиксированы общие ценности и правила поведения, которые должны соблюдаться независимо от текущей задачи. Файл `IDENTITY.md` дополняет эту информацию, предоставляя более конкретные данные о роли и имени агента, что помогает модели лучше понимать свой контекст [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Инструкции по взаимодействию с другими агентами хранятся в `AGENTS.md`, что позволяет разделить логику коммуникации от логики выполнения задачи [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Политики использования инструментов, которые являются критически важными для безопасности и корректности работы, детально прописываются в `TOOLS.md` [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Это позволяет иметь единый источник правды для правил вызова каждой функции. `USER.md` содержит актуальный контекст задачи, предоставленный пользователем, а `MEMORY.md` отслеживает историю диалога, обеспечивая последовательность и память [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Все эти файлы динамически подставляются в контекст модели на каждом новом шаге, что позволяет агенту постоянно оперировать актуальной информацией [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)].

Manus AI, хотя и является закрытой системой, демонстрирует другой аспект архитектурной декомпозиции через свою трехагентную модель: **Planner**, **Execution**, и **Verification** [[5](https://arxiv.org/html/2505.02024v2)]. Эта архитектура напрямую транслируется в специализацию промптов для каждого типа агента. Промпт для **Planner Agent** сфокусирован на высоком уровне стратегии и планирования. Его задача — разбивать сложные запросы пользователя на конкретные, выполнимые подзадачи и формировать пошаговый план [[5](https://arxiv.org/html/2505.02024v2)]. Промпт для **Execution Agent** должен содержать четкие инструкции по вызову инструментов для выполнения одного шага из плана [[5](https://arxiv.org/html/2505.02024v2)]. Наконец, **Verification Agent** получает промпт, ориентированный на качество и проверку. Он анализирует результаты выполнения, сравнивает их с ожидаемыми и принимает решение о необходимости исправления ошибки или перепланирования [[5](https://arxiv.org/html/2505.02024v2)]. Такое разделение ролей позволяет значительно повысить надежность системы, так как каждая часть процесса выполняется специализированным агентом, что снижает вероятность ошибок, характерных для универсальных систем. Хотя точные тексты промптов Manus недоступны, сам принцип его архитектуры является мощным ориентиром для проектирования собственных систем [[5](https://arxiv.org/html/2505.02024v2)].

Для проекта `agentsTG`, который использует стек aiogram + LangGraph с 7 агентами, анализ этих подходов имеет прямое практическое значение. Текущая структура с `souls/*.md` и `behavior.py` уже заложена в правильном направлении, но может быть оптимизирована. Рекомендуется рассмотреть возможность более четкого разделения обязанностей между файлами, подобно OpenClaw. Например, `IDENTITY.md` может содержать только чисто описательную информацию об агенте (имя, роль, основная функция), тогда как вся логика взаимодействия с другими агентами (например, правила делегирования) должна быть вынесена в отдельный файл, такой как `ROLES_AND_DELEGATION.md`. Это позволит избежать дублирования информации и усложнения `SOUL.md`. Кроме того, архитектура Manus подсказывает, что текущие 7 агентов можно сгруппировать по ролям. Возможно, один из них может выполнять функцию "Супервизора" или "Планировщика", который распределяет задачи между остальными "Специалистами" (Исполнители). Это упростит оркестрацию и сделает систему более управляемой.

В заключение, успешная архитектура системных промптов зависит от способности эффективно декомпозировать знания и правила. OpenClaw предлагает готовую модель модульности через файлы, в то время как Manus демонстрирует силу архитектурной специализации агентов. Проекту `agentsTG` следует взять за основу идею модульной сборки промптов, предложенную в OpenClaw, и применить принцип специализации ролей, заимствованный из архитектуры Manus, для дальнейшей оптимизации своей системы из семи агентов.

## Поведенческие директивы и управление потоком: Пробуждение, Перепланирование и Завершающий этап

Эффективность и автономность мультиагентной системы во многом определяются ее способностью управлять своим внутренним циклом работы, а также четко информировать пользователя о состоянии выполнения задачи. Анализ эталонных систем выявляет несколько ключевых механизмов, направленных на решение этих проблем: регулярные активации для автономного выполнения («пробуждающие сигналы»), логика перепланирования в случае сбоев и стандартизированные процедуры завершения задачи с фокусом на пользовательский опыт.

OpenClaw предоставляет конкретные инструменты для реализации автономного поведения через шаблоны «пробуждающих сигналов» (`proactive wake prompt templates`) [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Эти шаблоны предназначены для ситуаций, когда агенту необходимо выполнить задачу по расписанию или по событию, не дожидаясь нового запроса от пользователя. Например, агент-персональный ассистент может использовать такой шаблон для проверки календаря на предмет предстоящих встреч или напоминаний, которые нужно установить [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Включение таких инструкций в `SOUL.md` или отдельный файл конфигурации позволяет агенту действовать инициативно, что является значительным шагом в сторону повышения его автономности. В контексте проекта `agentsTG`, где есть агенты с функциями вроде `schedule_reminder`, внедрение подобного механизма является крайне актуальным. Вместо того чтобы постоянно ждать команд от пользователя, агент может периодически «просыпаться» и выполнять свои задачи самостоятельно, что делает взаимодействие с системой более бесшовным и предсказуемым.

Логика управления потоком, особенно способность системы к адаптации, является еще одним критически важным аспектом. Архитектура Manus AI, основанная на цикле «план → выполнение → перепланирование», является классическим примером продуманного управления потоком [[5](https://arxiv.org/html/2505.02024v2)]. Если Execution Agent сообщает о неудаче или если Verification Agent обнаруживает несоответствие результата ожиданиям, весь процесс не аварийно завершается. Вместо этого система возвращается к Planner Agent, который пересматривает план и предлагает новый путь решения задачи [[5](https://arxiv.org/html/2505.02024v2)]. Хотя детальные промпты для этой логики в Manus недоступны, сам принцип подчеркивает необходимость наличия в системе `agentsTG` специальных директив для оркестратора. Вместо простого делегирования задачи, оркестратор должен уметь принимать решения о продолжении, завершении или изменении стратегии. Это требует расширения текущего `ORCHESTRATOR_JSON_DIRECTIVE` в `behavior.py` для включения новых типов действий, таких как `request_replan` или `checkpoint_status`. Это позволит системе быть более устойчивой к неопределенности и ошибкам, характерных для сложных задач.

Финализация задачи и представление результатов пользователю — это заключительный, но не менее важный этап. Качество этого взаимодействия напрямую влияет на восприятие всей системы. Здесь также можно почерпнуть идеи у Manus, который, согласно доступной информации, уделяет внимание пользовательскому опыту прогресса, таким как отображение шагов («шаг N из M») и скрытие промежуточных рассуждений агента [[5](https://arxiv.org/html/2505.02024v2)]. Эта практика помогает пользователю следить за ходом выполнения сложной задачи, не перегружая его техническими деталями и промежуточными выводами. Для проекта `agentsTG` это означает, что текущая логика `finalize pass` в `assembler.py` может быть значительно улучшена. Вместо того чтобы просто объединять все ответы агентов, следует создать специальный механизм («Finalizer»), который будет формировать единое, структурированное сообщение для пользователя. Этот финальный ответ должен включать краткое резюме выполненных действий, сами результаты (артефакты) и, возможно, ссылки на более подробную информацию. Использование HTML-разметки для Telegram, как указано в запросе, позволит сделать это сообщение визуально привлекательным и легко читаемым. Явная инструкция для агента-финализатора о том, что делать с промежуточными данными, и как представлять конечный результат, станет ключевым элементом улучшения пользовательского опыта.

В совокупности, внедрение этих механизмов управления потоком и поведенческими директивами позволит системе `agentsTG` стать более зрелой и автономной. Регулярные активации позволят агентам действовать инициативно, логика перепланирования сделает систему более отказоустойчивой, а продуманный финализирующий этап обеспечит четкое и понятное взаимодействие с пользователем. Эти изменения потребуют доработки файлов `souls/*.md` для добавления режимов «пробуждения», расширения `behavior.py` для поддержки новых типов действий оркестратора и рефакторинга `assembler.py` для создания более мощного финализирующего модуля.

## Механизмы безопасности и политики инструментов

Безопасность является краеугольным камнем любой архитектуры агентных систем, способных выполнять действия в реальном мире. Анализ производственных систем, таких как Manus и OpenClaw, выявляет три ключевых механизма защиты, которые широко применяются в отрасли: одобрение во время выполнения, спецификация политик и конфигурация границ. Эти механизмы часто используются в комбинации, поскольку ни один из них по отдельности не является полностью надежным. Особенно важно отметить, что многие исследователи и практики переходят от reliance на LLM-самоисполнение правил к использованию детерминированных программных проверок, что значительно повышает уровень защиты.

Одобрение во время выполнения является самым распространенным механизмом безопасности, используемым в 15 из 21 проанализированных производственных систем [[4](https://arxiv.org/html/2605.24309v1)]. Суть этого механизма заключается в том, что перед выполнением любого потенциально опасного действия (например, удаление файла, отправка платежа, запись в базу данных) агент должен получить явное разрешение от пользователя [[4](https://arxiv.org/html/2605.24309v1)]. Однако, несмотря на широкое распространение, этот подход имеет существенный недостаток — он создает высокую когнитивную нагрузку на пользователя, известную как «усталость от подтверждения» [[4](https://arxiv.org/html/2605.24309v1)]. В одном из случаев, задача по рефакторингу кода могла генерировать десятки отдельных запросов на подтверждение операций с файлами, что приводило к частым контекстным переключениям со стороны пользователя и снижало общую продуктивность [[4](https://arxiv.org/html/2605.24309v1)]. Поэтому наиболее разумная стратегия заключается в выборочном применении этого механизма. Системы, ориентированные на разработчиков, чаще всего запрашивают подтверждение для любых операций с файлами [[8](https://arxiv.org/html/2602.17753v1)]. Для проекта `agentsTG` это означает, что не стоит запрашивать подтверждение для каждого вызова инструмента. Вместо этого следует провести категоризацию инструментов по степени риска и применять запрос подтверждения только для самых опасных из них, например, связанных с изменением данных или внешними платежами.

Второй по популярности механизм — спецификация политик, которая используется в 14 из 21 систем [[4](https://arxiv.org/html/2605.24309v1)]. Он заключается в определении четких правил, которым агент должен следовать при использовании своих инструментов. Проблема возникает, когда эти правила задаются в виде обычного текста внутри системного промпта (естественно-языковые политики). В этом случае исполнение правил сводится к самоисполнению LLM, что делает систему крайне уязвимой для атак путем внедрения инструкций, так как злоумышленник может попытаться переубедить самого же агента нарушить его же правила [[4](https://arxiv.org/html/2605.24309v1)]. Более надежным подходом является использование детерминированных программных проверок. Примером такого подхода является библиотека CogniWall, которая действует как промежуточное звено между LLM и исполняемым инструментом [[1](https://dev.to/cogniwall/securing-langgraph-multi-agent-workflows-how-to-enforce-tool-level-permissions-13cm)]. После того как LLM сгенерирует предполагаемый вызов инструмента, CogniWall выполняет быстрые и детерминированные проверки (например, проверка суммы платежа на соответствие лимиту, поиск персональных данных в аргументах) [[1](https://dev.to/cogniwall/securing-langgraph-multi-agent-workflows-how-to-enforce-tool-level-permissions-13cm)]. Только если вызов проходит все программные проверки, он передается на выполнение. Если проверка не проходит, CogniWall возвращает агенту подробное описание причины блокировки, позволяя ему скорректировать свое поведение, вместо того чтобы аварийно завершать работу. Для проекта `agentsTG` текущая логика в `tier_rules.py` и `deny lists` является хорошим шагом в правильном направлении. Однако, чтобы повысить безопасность, следует рассмотреть внедрение аналогичного middleware-слоя, который бы проверял каждый вызов инструмента перед его фактическим выполнением. Это позволит отделить логику безопасности от промптов и сделать ее более надежной.

Третий механизм — конфигурация границ, который заключается в предоставлении каждому агенту только ограниченного набора инструментов и прав, необходимых для выполнения его конкретной роли [[8](https://arxiv.org/html/2602.17753v1)]. Это соответствует принципу минимальных привилегий. Однако даже здесь существует риск, известный как «расширение границы», когда разрешения, предоставленные на начальном этапе, могут только расти, но не сужаться в течение одной сессии [[4](https://arxiv.org/html/2605.24309v1)]. Это привело, например, к инциденту с удалением базы данных в Replit Agent, который был вызван слишком широкой конфигурацией песочницы [[4](https://arxiv.org/html/2605.24309v1)]. Большинство систем поддерживают только расширяемость, но лишь два из 21 проанализированных production system поддерживали возможность сужения области действия [[4](https://arxiv.org/html/2605.24309v1)]. Наши `tier` (LIGHT/STANDARD/FULL) уже являются отличным механизмом управления областью действия. Необходимо убедиться, что эта логика не только определяет набор инструментов при создании агента, но и может быть динамически изменена, если это требуется для безопасности. Также критически важно внедрить механизм аудита, где каждое действие сопровождается метаданными о том, кто его выполнил и почему (подлинность), чтобы обеспечить прозрачность и возможность расследования инцидентов [[9](https://arxiv.org/html/2603.26221v1)].

Наконец, важно знать о распространенных антипаттернах, которых следует избегать. Одним из них является «ложное обещание инструмента» — обещание модели выполнить любую задачу, что противоречит принципу прозрачности [[11](https://arxiv.org/html/2603.10060v1), [12](https://arxiv.org/pdf/2603.10060)]. Другой антипаттерн — «эхо» или повторение информации из контекста без добавления нового значения, что неэффективно и раздражает пользователя. Наконец, исследования показали, что некоторые защитные меры, такие как добавление явных правил безопасности в промпт или обертывание внешнего контента в специальные маркеры, оказываются неэффективными против продвинутых атак, таких как внедрение плана, где злоумышленник инжектирует в план агента скрытые вредоносные инструкции [[7](https://arxiv.org/html/2506.17318v1)]. Это подчеркивает, что защита должна быть многоуровневой и включать как промпты, так и надежные программные проверки.

## Шаблоны и концепции из open-source фреймворков

Помимо анализа коммерческих систем, изучение открытых фреймворков, таких как LangGraph, AutoGPT, CrewAI и OpenHands, предоставляет ценные идеи и шаблоны для проектирования промптов, которые можно адаптировать в проекте `agentsTG`. Эти фреймворки, будучи результатом коллективной работы сообщества, часто служат хранилищем лучших практик в области агентных систем, особенно в части декомпозиции задач, ролевой игры и организации взаимодействия между агентами.

LangGraph, как фреймворк для построения графовых workflow, предоставляет мощные инструменты для оркестрации, но не навязывает жестких правил для структуры промптов. Тем не менее, официальные примеры демонстрируют хорошие практики для промптов супервизора. В таких промптах обычно содержатся инструкции по выбору следующего узла в графе, что напрямую связано с `ORCHESTRATOR_JSON_DIRECTIVE` в проекте `agentsTG`. Супервизору дается описание всех доступных агентов и их возможностей, и его задача — на основе запроса пользователя выбрать наиболее подходящий путь в графе. Это соответствует архитектуре, где один центральный агент (супервизор) координирует работу группы специалистов. Интеграция LangGraph и CrewAI, где CrewAI отвечает за роль и задачи агентов, а LangGraph — за управление состоянием и потоком, является мощной комбинацией, которую можно эмулировать в рамках одного проекта [[3](https://arxiv.org/html/2411.18241v1)].

AutoGPT и CrewAI делают особый акцент на двух ключевых аспектах: создании персонажа и политике использования инструментов. В CrewAI, например, агенты создаются с четко определенными ролями, целями и инструментами [[3](https://arxiv.org/html/2411.18241v1)]. Это позволяет им выступать в качестве автономных единиц, способных принимать решения и взаимодействовать друг с другом. Промпты для таких агентов включают описание их персонажа (например, "Ты — опытный исследователь, который любит глубоко погружаться в тему") и список доступных инструментов с их назначением. Эта концепция полностью соответствует текущей структуре `agentsTG` с файлами `souls/*.md` (для описания персонажа) и `behavior.py` (для определения доступных инструментов и правил их использования). Изучение примеров из репозиториев AutoGPT и CrewAI может дать множество идей для улучшения формулировок в этих файлах, сделав промпты более конкретными и эффективными.

OpenHands представляет собой интересный случай, поскольку он предназначен для развития агентов, взаимодействующих с миром подобно человеку-разработчику, выполняя действия в песочнице [[2](https://openreview.net/forum?id=OJd3ayDDoF)]. Хотя промпты для OpenHands в открытом доступе ограничены, сама платформа подчеркивает важность безопасного выполнения кода и взаимодействия с командной строкой. Это напрямую относится к безопасности в `agentsTG` и подчеркивает необходимость использования изолированных сред (песочниц) для выполнения инструментов, особенно тех, которые могут выполнять код или изменять файловую систему. Поддержка различных LLM и возможность интеграции с различными агентами делают OpenHands гибким инструментом для тестирования идей [[2](https://openreview.net/forum?id=OJd3ayDDoF)].

Что касается агентов от Claude (Claude Code / Cursor), анализ показывает, что они используют простые, но эффективные файлы конфигурации, такие как `.cursorrules`, для определения политик [[4](https://arxiv.org/html/2605.24309v1)]. Это указывает на то, что для некоторых политик нет необходимости создавать сложные многострочные промпты. Декларативный подход, где политики задаются в виде простых правил в YAML- или Markdown-файлах, может быть легче поддерживать и анализировать. Это может быть хорошей альтернативой или дополнением к `tier_rules.py` в `agentsTG`, позволяя отделить политику от кода и упрощая управление доступом.

В итоге, открытые фреймворки предлагают богатый набор идей для улучшения `agentsTG`. Можно заимствовать шаблоны промптов супервизора из LangGraph, концепцию ролевой игры и автоматического делегирования из CrewAI, идеи по безопасности и изоляции из OpenHands, и, возможно, декларативный подход к политикам из практики Cursor. Эти идеи можно и нужно адаптировать к существующей архитектуре на базе LangGraph, чтобы сделать систему более мощной, безопасной и удобной для использования.

## Сравнительный анализ и рекомендации по адаптации для agentsTG

На основе всестороннего анализа эталонных архитектур и лучших практик, представленных в источниках, можно сформулировать конкретные рекомендации по улучшению системного слоя запросов в проекте `agentsTG`. Цель состоит в том, чтобы повысить автономность, безопасность и надежность системы, сделав ее поведение более предсказуемым и управляемым. Анализ выявил как сильные стороны текущей реализации, так и ряд областей для доработки.

**Сильные стороны текущей реализации `agentsTG`:**
1.  **Структура на основе файлов:** Использование `souls/*.md` для определения идентичности и `behavior.py` для директив является хорошей отправной точкой, соответствующей модульному подходу, популяризованному OpenClaw [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)].
2.  **Механизм тарифов:** Логика `tier` (LIGHT/STANDARD/FULL) для управления доступом к инструментам является правильным шагом в сторону контроля над областью действия, что соответствует лучшим практикам [[8](https://arxiv.org/html/2602.17753v1)].
3.  **Прямолинейный оркестратор:** Использование LangGraph в качестве оркестратора предоставляет гибкость для построения сложных рабочих процессов, а текущая директива `ORCHESTRATOR_JSON_DIRECTIVE` начинает решать задачу маршрутизации [[3](https://arxiv.org/html/2411.18241v1)].

**Выявленные пробелы и рекомендации по их устранению:**

**1. Декомпозиция Identity/SOUL и Behavior:**
*   **Пробел:** В файле `services/prompts/behavior.py` смешаны общие системные директивы (`GOAL_DIRECTIVE`) и специфические для агента стили (`MANUS_PA_STYLE`). Это затрудняет поддержку и масштабирование, так как общие правила могут случайно измениться для определенного агента.
*   **Рекомендация:** Необходимо провести декомпозицию. Общие директивы, такие как `GOAL_DIRECTIVE`, следует вынести в отдельный, возможно, общий файл (например, `system_directives.md`). Стилистические директивы, специфичные для агента (например, стиль персонального ассистента), остаются в `behavior.py`, но могут быть вынесены в отдельные файлы для каждого агента (например, `pa_style.md`), чтобы избежать мутации. Также следует рассмотреть создание файла `IDENTITY.md` для вынесения статической информации об агенте (имя, роль, основная сфера ответственности), чтобы `SOUL.md` остался чистым и сфокусированным на философии и ценностях [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)].

**2. Отсутствие внешней безопасности и надежной политики инструментов:**
*   **Пробел:** Вероятно, логика безопасности, определенная в `tier_rules.py` и списках запретов, выполняется внутри LLM. Это уязвимо, так как LLM-самоисполнение правил легко обходится с помощью атак путем внедрения инструкций [[4](https://arxiv.org/html/2605.24309v1)].
*   **Рекомендация:** Самая важная рекомендация по безопасности — внедрить программный слой проверки (промежуточное ПО) для вызовов инструментов. Этот слой должен находиться между генерацией вызова LLM и его фактическим выполнением. Он будет проверять каждый вызов на соответствие детерминированным правилам (например, финансовый лимит, наличие PII, принадлежность инструмента к разрешенному тарифу) [[1](https://dev.to/cogniwall/securing-langgraph-multi-agent-workflows-how-to-enforce-tool-level-permissions-13cm)]. Это соответствует лучшей практике, описанной в работе о secure-by-construction agent platforms [[9](https://arxiv.org/html/2603.26221v1)]. Реализация такого механизма значительно повысит надежность системы и позволит удалить часть этой логики из промптов, сделав их проще и безопаснее.

**3. Нечеткая структура внешнего цикла и логика перепланирования:**
*   **Пробел:** Текущая директива `ORCHESTRATOR_JSON_DIRECTIVE` в `behavior.py` скорее всего, фокусируется на делегировании задач, но не содержит явной логики для контрольных точек, перепланирования или завершения задачи.
*   **Рекомендация:** Необходимо расширить директиву для поддержки более сложных сценариев. В `behavior.py` следует добавить новую директиву, например, `REPLAN_DIRECTIVE`, которая будет содержать инструкцию для оркестратора, что делать, если план провалился или результат неудовлетворителен. Это соответствует архитектуре Manus AI, где есть отдельный этап проверки и перепланирования [[5](https://arxiv.org/html/2505.02024v2)]. Также стоит ввести ограничение на максимальное количество шагов (`MAX_TURNS`) для предотвращения бесконечных циклов.

**4. Пробелы в Finalize Pass и UX:**
*   **Пробел:** Логика объединения результатов от 7 агентов, вероятно, реализована достаточно просто, без учета пользовательского опыта.
*   **Рекомендация:** Необходимо создать специальный "Finalizer" агента или функцию в `assembler.py`. Его задача — получать все результаты, анализировать их и формировать единое, последовательное и структурированное сообщение для пользователя. Это сообщение должно включать краткое резюме выполненных действий, сами артефакты (ссылки, файлы) и, возможно, ссылки на детали выполнения. Это улучшит UX, сделав результаты понятными и легко доступными. Идея "скрытия мыслей" из Manus AI напрямую применима: промежуточные рассуждения агентов должны отправляться в секретном режиме или вообще не показываться пользователю [[5](https://arxiv.org/html/2505.02024v2)].

**5. Автономность и Proactive Behavior:**
*   **Пробел:** Система в основном реактивна и ждет команд от пользователя.
*   **Рекомендация:** Для агентов, которым необходима автономность (например, агент, который должен устанавливать напоминания), следует добавить в их `SOUL.md` или отдельный файл конфигурации специальный блок с инструкциями для "пробуждающего сигнала" или режима "по расписанию" [[14](https://ar5iv.labs.arxiv.org/html/2603.17419)]. Это позволит им выполнять задачи по расписанию без постоянного вмешательства пользователя.

Ниже представлены черновые фрагменты на русском языке, адаптированные под требования проекта `agentsTG`, с указанием изменений по сравнению с текущей реализацией.

### **E.1. MANUS_PA_STYLE для Эльзы (автономность, schedule_reminder, без false promises)**

```markdown
# Эльза (персональный ассистент)

## Цель
Твоя задача — помочь пользователю организовать его день, управлять напоминаниями и выполнять другие административные задачи. Ты работаешь автономно.

## Правила
1.  **Автономность**: Старайся выполнять задачи самостоятельно. Не проси дополнительных данных у пользователя, если можешь найти их самостоятельно или если это не критично.
2.  **Напоминания**: Используй только инструмент `schedule_reminder`. Если пользователь просит о чем-то другом, уточни, имея в виду напоминание.
3.  **Запретные возможности**: Ты не можешь и не должен обещать выполнение задач, которые выходят за рамки твоих возможностей. Твоя сфера — организация и напоминания.
4.  **Прогресс**: Если ты выполняешь длительную задачу, периодически информируй о своем состоянии (например, "Записываю напоминание...").

## Режим «Пробуждения»
Если ты не получаешь команд в течение часа, проверь календарь пользователя на предстоящие события и напоминания, которые нужно установить.
```

**Что меняется vs текущий behavior.py:**
*   Добавлена четкая директива об автономности (`AVOID_USER_INTERACTION`).
*   Удалены любые намеки на универсальность (антимодель "ложное обещание").
*   Явно указан инструмент `schedule_reminder`.
*   Добавлено правило для "пробуждающего сигнала" (`HEARTBEAT`).

### **E.2. ORCHESTRATOR_JSON_DIRECTIVE (supervisor + plan + direct_reply)**

```python
"""
Ты — Супервизор. Твоя задача — принимать решения о том, какой агент должен выполнить задачу.
Верни JSON с одним из следующих полей:
- 'action_type': 'delegate', 'final_answer' или 'direct_reply'
- 'agent_name': Имя агента, которому поручить задачу (только при action_type='delegate')
- 'task_description': Описание задачи для агента (только при action_type='delegate')
- 'final_answer': Твой итоговый ответ для пользователя (только при action_type='final_answer')
- 'user_message': Сообщение для пользователя (только при action_type='direct_reply')

Правила принятия решения:
1.  Если задача пользователя относится к конкретному агенту (например, "Найди статью"), выбери этого агента.
2.  Если задача требует синтеза информации от нескольких агентов, выбери "Супервизор".
3.  Если задача простая и ты знаешь ответ, верни его в 'final_answer'.
4.  Если задача требует немедленного ответа, не связанный с инструментами, используй 'direct_reply'.
5.  Обязательно добавь краткое обоснование своего выбора в поле 'reasoning'.

Пример ответа для делегирования:
{{
  "action_type": "delegate",
  "agent_name": "ResearchAgent",
  "task_description": "Найди последние новости о развитии искусственного интеллекта.",
  "reasoning": "Задача относится к области исследования, поэтому делегирую ее ResearchAgent."
}}

Пример ответа для финального ответа:
{{
  "action_type": "final_answer",
  "final_answer": "Я нашел нужную информацию. Вот она...",
  "reasoning": "Задача была простой и требовала одного факта, который я могу предоставить сразу."
}}
"""
```

**Что меняется vs текущий behavior.py:**
*   Расширена структура JSON для включения поля `action_type`.
*   Добавлен тип `direct_reply` для прямых ответов.
*   Добавлено поле `reasoning` для логики принятия решений.

### **E.3. FINALIZE_USER_REPLY после tool calls**

```markdown
# Финализатор для пользователя

## Цель
Создать итоговое сообщение для пользователя, которое суммирует результаты выполнения задачи всеми агентами и представляет артефакты.

## Структура ответа
Твоя задача — сгенерировать HTML-сообщение для Telegram по следующей структуре:

<h2>✅ Задача выполнена!</h2>
<p><b>Краткое резюме:</b> {краткое описание выполненных действий}</p>

<h3>📄 Результаты:</h3>
<ul>
{артефакты}
</ul>

<h3>🔍 Детали выполнения:</h3>
```
